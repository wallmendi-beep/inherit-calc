# logic_map.md

검산 AI와 구현자가 현재 제품 동작을 빠르게 파악하기 위한 로직 맵이다.  
이 문서는 **계산 엔진만이 아니라 제품 동작 전체**를 기준으로 작성한다.

---

## 1. 계층 구조

현재 앱의 핵심 계층은 다음과 같다.

| 계층 | 파일 | 역할 |
|---|---|---|
| 공통 유틸 | `utils.js` | 법 시기 판정, 날짜 비교, 분수 계산 |
| 분배 엔진 | `inheritance.js` | 상속권 판정, 대습/재상속, 차순위 자동 탐색, 지분 계산 |
| 사후 검증 | `inheritanceAudit.js` | 최종 결과 무결성 검사 |
| 트리 정규화 | `../utils/treeDomain.js` | import 데이터 정규화, `personId`/중복/기본값 처리 |
| 앱 액션 | `../utils/appActions.js` | AI 입력, 파일 불러오기/저장, import 후 이동 정책 |
| 스마트 가이드 | `../hooks/useSmartGuide.js` | 경고를 사용자 행동으로 연결 |
| 시나리오 검증 | `check_logic.js` | 사람이 읽는 예시 중심 검증 스크립트 |

## 2. 제품 동작 흐름

### 2.1 입력 / import

- 사용자가 직접 입력하거나
- JSON 파일을 불러오거나
- AI JSON을 주입한다.

이 단계에서 `treeDomain.js`가:

- 허용 필드만 보존하고
- `personId`와 `id`를 정리하며
- 동일 `personId` 반복 등장은 상속 제외가 아니라 하위 전개 중단으로 처리하며
- 선사망 배우자/선사망 자녀의 기본 제외 상태를 정규화한다.

### 2.2 계산 준비

- `utils.js`가 법 시기와 날짜 비교를 담당한다.
- `inheritance.js`는 현재 트리를 기준으로 상속인 후보를 고른다.

### 2.3 계산

`inheritance.js`의 큰 순서는 다음과 같다.

1. 현재 노드의 사망 시점과 기준 사망일 비교
2. 선사망/후사망/결격/상실 여부 판정
3. `isRenounced()`로 제외 대상을 걸러냄
4. 대습 가지 유효성 판정
5. 필요 시 `findGlobalSuccessors()`로 차순위 탐색
6. 법 시기별 가중치 계산
7. 분수 합산 및 최종 그룹 정리

### 2.4 검산

`inheritanceAudit.js`는:

- 총합 불일치
- 미전달 중간 지분
- 사망자 지분 잔존
- 구조 오류

를 검사해 결과 무결성을 평가한다.

### 2.5 사용자 피드백

`useSmartGuide.js`는:

- 입력 누락
- 중복 배우자
- 후속 상속인 확인
- 호주 검토
- 차순위 여성 상속인 검토

등을 `mandatory` / `recommended` 가이드로 생성한다.

## 3. 계산 엔진과 제품 동작의 구분

이 문서에서 특히 구분해야 할 두 층이 있다.

### 3.1 계산 엔진

- 법 시기 결정
- 상속권·대습권 판정
- 지분 분배
- 최종 분수 계산

주 파일:

- `inheritance.js`
- `utils.js`

### 3.2 제품 동작

- import 정규화
- 탭 이동
- 경고/가이드 생성
- audit 결과 노출

주 파일:

- `treeDomain.js`
- `appActions.js`
- `useSmartGuide.js`
- `TreePanel.jsx`, `CalcPanel*`, `ResultPanel*`

즉, 제품은 “엔진 계산기”가 아니라 “입력 정리 + 계산 + 검산 + 가이드”의 조합이다.

## 4. 현재 구현에서 중요한 로직 포인트

### 4.1 `successorStatus`

- 사용자가 후속 상속인 없음 등을 확정하면 자동 탐색을 멈추는 데 사용된다.
- 따라서 엔진은 항상 “하위가 비었는가”만 보지 않고, “사용자가 이미 확정했는가”도 함께 본다.

### 4.2 `personId`

- 동일인 연결의 기준 필드다.
- 다만 `personId`가 없더라도 `id` 또는 대체 키로 대습 그룹이 분리되어야 한다.
- 동일 `personId`가 여러 사건 단계에 반복 등장하는 것은 정상적인 재상속/대습 흐름일 수 있다.
- 따라서 반복 등장 자체를 `duplicate` 상속 제외 사유로 바꾸지 않는다.
- 반복 등장 노드는 하위 `heirs` 전개만 중단해 순환·중복 순회를 막고, 해당 사건에서의 상속인 자격은 사건 기준일과 관계에 따라 별도로 판정한다.

### 4.2.1 동명이인 표기

- 실제 동명이인은 이름 자체를 `김경희(A)`, `김경희(B)`처럼 구분한다.
- 같은 사람의 반복 등장은 같은 `personId`와 같은 구분 이름을 유지한다.
- 검산표와 사건보고서에서는 이름만 보고도 사람을 구분할 수 있어야 하므로, 동명이인을 같은 `name`으로 방치하지 않는다.

### 4.3 `isPrimaryHojuSuccessor`

- 호주 가산을 사람이 명시적으로 제어할 수 있는 핵심 상태다.

### 4.4 무효 대습 가지

- 하위 상속인이 있어도 모두 무자격이면 대습 가지는 성립하지 않는다.
- 대표 예: 1991년 이전 선사망 딸 아래 사위만 있는 경우
- 이때는 가지를 무효 처리하고 원단계에서 재산정한다.

### 4.5 선사망·후속없음 분배대상 제외

- 현재 사건 기준일보다 먼저 사망한 인물이 유효한 대습/후속 상속인을 갖지 않으면 분배분모에 포함하지 않는다.
- `isExcluded: true`, `exclusionOption: "predeceased"` 상태라도 하위 구조가 있는 경우에는 대습상속 가능성을 판정하고, 하위 구조가 없거나 모두 무자격이면 해당 인물의 가중치는 0으로 처리한다.
- 반복 등장 동일인의 하위 상속인을 이름 또는 `personId`로 빌려오는 경우에도, 빌려온 후의 유효 상속인 여부를 기준으로 분배대상 포함 여부를 재판정한다.
- 김혁조 검산 기준으로 `김명남` 사건의 `윤우경`, `윤우철`처럼 선사망했고 후속 상속인이 없는 인물은 분배분모에서 제외한다.

## 5. Audit와 Guide의 관계

둘은 같은 것이 아니다.

- `inheritanceAudit.js`는 결과를 검사한다.
- `useSmartGuide.js`는 사용자의 다음 행동을 제안한다.

예:

- `final-total-mismatch`는 audit 이슈
- “후속 상속인을 입력해 주세요”는 guide
- `ineligible-substitution-heirs`는 engine warning이며, 필요 시 guide에 연결될 수 있다

---

## 6. `check_logic.js`의 현재 위치

`check_logic.js`는 핵심 엔진 그 자체가 아니라:

- 사람이 읽는 예시 출력
- 선례 시나리오 확인
- 최근 회귀 포인트 육안 점검

용 스크립트다.

따라서 구현 기준의 SSOT는 아니며, 엔진 설명 문서에서는 보조 검증 도구로 취급한다.

---

## 7. 문서 유지 원칙

- 엔진 구조가 바뀌면 이 문서도 같이 갱신한다.
- `check_logic.js`가 바뀌었다고 엔진이 바뀐 것으로 보지 않는다.
- 이 문서는 “검산 AI 참조용”이지만, 실제 코드와 다른 설명을 두지 않는다.
