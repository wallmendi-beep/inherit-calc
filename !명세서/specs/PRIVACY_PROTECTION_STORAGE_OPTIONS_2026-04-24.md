# 개인정보 보호 저장 방식 설계 노트 (2026-04-24)

## 1. 배경

상속 사건 저장 파일은 현재 JSON 기반이라 메모장 등 일반 프로그램으로 열었을 때 내용을 읽을 수 있다.  
이 특성은 디버깅과 호환성에는 유리하지만, 주민등록번호·가족관계·사망일자·혼인 이력 등 민감한 개인정보가 포함될 수 있다는 점에서 보안상 약점이 있다.

이번 문서는 다음 요구를 기준으로 경우의 수를 정리한다.

- 메모장 등 일반 프로그램에서는 내용을 바로 읽기 어렵게 만들기
- 가능하면 이 계산기에서는 추가 입력 없이 자연스럽게 열리게 만들기
- 개인정보 유출 위험을 낮추되, 구현 복잡도와 운영 리스크도 함께 고려하기

## 2. 핵심 전제

### 2-1. 비밀번호 없는 완전한 보호는 어렵다

사용자가 비밀번호를 입력하지 않고도 프로그램이 자동으로 열 수 있으려면, 복호화에 필요한 키가 결국 프로그램 또는 실행 환경 안에 존재해야 한다.  
따라서 다음 조합은 완전한 보안을 제공할 수 없다.

- 메모장에서는 안 보임
- 사용자 비밀번호 입력 없음
- 이 프로그램에서는 자동 열림

이 조합에서 가능한 것은 아래 두 가지다.

- `노출 억제`: 일반 사용자가 파일을 열어도 내용을 바로 읽기 어렵게 만들기
- `환경 종속 보호`: 특정 PC/사용자 계정에서만 자동 복호화되게 만들기

## 3. 목표 수준별 선택지

## 3-1. 옵션 A: 전용 포맷 + 난독화

### 개념

JSON을 그대로 저장하지 않고, Base64 인코딩·압축·커스텀 래핑 등을 적용한 전용 포맷으로 저장한다.

예시:

```json
{
  "type": "inheritance-case-obfuscated",
  "version": "v1",
  "payload": "BASE64_OR_COMPRESSED_TEXT"
}
```

### 장점

- 구현이 가장 쉽다
- 기존 브라우저 앱 구조에서 바로 적용 가능하다
- 메모장으로 열어도 일반 사용자가 내용을 바로 읽기 어렵다
- 기존 저장/불러오기 UX를 크게 바꾸지 않아도 된다

### 단점

- 엄밀한 보안이 아니다
- 로직을 알면 복원 가능하다
- 공격자 관점에서는 “난독화”이지 “암호화”가 아니다

### 적합한 목적

- 메모장 즉시 노출 방지
- 비전문가의 우발적 열람 억제

### 구현 방법

1. `serializeFactTree()` 또는 `saveFactTreeToFile()` 단계에서 JSON 문자열 생성
2. 문자열을 `CompressionStream` 또는 단순 압축 라이브러리로 압축
3. 결과를 Base64로 인코딩
4. `.json` 대신 `.icase` 같은 전용 확장자 사용
5. `loadTreeFromJsonFile()`에서 `type` 필드로 포맷 판별 후 자동 복원

### 권장도

`낮은 비용으로 노출을 줄이는 1차 방어`로는 적절하다.

## 3-2. 옵션 B: 프로그램 내장 키 기반 자동 암호화

### 개념

프로그램 내부에 고정 키를 두고 파일을 암호화/복호화한다.

### 장점

- 메모장에서는 사실상 평문이 보이지 않는다
- 사용자는 비밀번호를 입력하지 않아도 된다

### 단점

- 키가 프로그램 안에 있으므로 역공학에 취약하다
- 앱을 분석하면 결국 복호화 가능하다
- “이 프로그램에서만 열린다”는 느낌은 줄 수 있지만, 보안 강도는 기대보다 약하다

### 적합한 목적

- 형식적 보호
- 일반 사용자 수준의 열람 방지

### 구현 방법

1. 앱 코드에 고정 키 또는 키 파생 로직 내장
2. 저장 시 `AES-GCM`으로 암호화
3. 불러오기 시 자동 복호화
4. 파일 포맷에 `iv`, `ciphertext`, `version` 저장

### 권장도

`보안처럼 보이지만 실제 강도는 낮다`는 점 때문에 장기 권장안으로는 부적합하다.

## 3-3. 옵션 C: OS 계정 종속 암호화

### 개념

Windows 사용자 계정 또는 디바이스 보안 저장소에 묶인 키를 사용해 파일을 암호화한다.

예:

- Windows DPAPI
- OS credential store
- 브라우저만으로 어렵다면 별도 로컬 헬퍼/런처 연동

### 장점

- 사용자 비밀번호 입력 없이 자동 복호화 가능
- 메모장에서는 읽을 수 없다
- 다른 PC로 파일을 복사해도 같은 계정/환경이 아니면 열 수 없게 만들 수 있다
- 개인정보 보호 목적에 가장 부합한다

### 단점

- 현재 브라우저 중심 구조에서는 바로 구현하기 어렵다
- 네이티브 브리지 또는 로컬 앱 확장이 필요할 수 있다
- 기기 이전, 백업, 계정 변경 시 UX가 복잡해질 수 있다

### 적합한 목적

- 실질적 개인정보 보호
- “같은 사용자 환경에서만 자동으로 열린다”는 요구

### 구현 방법

1. 로컬 실행 환경이 브라우저만인지, 데스크톱 셸/런처 확장이 가능한지 확인
2. 가능하면 OS 종속 키 발급/복호화 helper 도입
3. 저장 파일 포맷은 다음처럼 구성

```json
{
  "type": "inheritance-case-protected",
  "version": "v1",
  "protection": "windows-user-bound",
  "ciphertext": "...",
  "metadata": {
    "caseNo": "...",
    "savedAt": "..."
  }
}
```

4. 동일 사용자/동일 장치에서 자동 복호화
5. 환경이 다르면 “이 장치에서 만든 보호 파일” 안내 후 열기 차단

### 권장도

`비밀번호 없이도 개인정보 보호 수준을 높이려면 가장 현실적인 장기안`이다.

## 3-4. 옵션 D: 사용자 비밀번호 기반 암호화

### 개념

사용자가 저장/불러오기 시 비밀번호를 입력하고, 그 비밀번호로 파일을 암호화한다.

### 장점

- 보안 강도가 가장 명확하다
- 파일 이동성과 독립성이 좋다
- 프로그램 외부에서 파일이 유출돼도 비밀번호 없이는 열기 어렵다

### 단점

- 사용자 경험이 번거롭다
- 비밀번호 분실 시 복구가 어렵다
- 현재 요구사항인 “이 프로그램에서는 자동으로 열리게”와 맞지 않는다

### 권장도

보안 강도는 가장 좋지만, 현재 사용자 요구와는 다소 다르다.

## 4. 비교 요약

| 옵션 | 메모장 노출 방지 | 사용자 추가 입력 | 보안 강도 | 구현 난이도 | 비고 |
|---|---|---:|---|---|---|
| A. 전용 포맷 + 난독화 | 중 | 없음 | 낮음 | 낮음 | 1차 방어용 |
| B. 내장 키 암호화 | 높음 | 없음 | 낮음~중간 | 중간 | 역공학 취약 |
| C. OS 계정 종속 암호화 | 높음 | 없음 | 중간~높음 | 높음 | 장기 권장 |
| D. 비밀번호 암호화 | 높음 | 있음 | 높음 | 중간 | 현재 요구와 불일치 |

## 5. 추천 전략

## 5-1. 단기

우선 `옵션 A`를 검토한다.

- 메모장 즉시 노출을 줄일 수 있다
- 기존 저장/불러오기 UX를 거의 유지할 수 있다
- “이 계산기 전용 파일처럼 보이게” 만드는 효과가 있다

단, 문서와 UI에 명확히 적어야 한다.

- 이것은 `보안 강화`라기보다 `우발적 노출 억제`다

## 5-2. 중기

환경 제약을 점검한 뒤 `옵션 C` 가능성을 검토한다.

- 데스크톱 앱/로컬 브리지 확장 가능 여부
- 동일 사용자 계정 종속 복호화가 가능한지
- 백업/이전 UX를 어떻게 처리할지

## 5-3. 비권장

현재 요구를 만족시키기 위해 `옵션 B`를 최종안으로 채택하는 것은 권장하지 않는다.

이유:

- 보안 기대치를 과하게 만들 수 있다
- 실제 보호 강도는 제한적이다

## 6. 구현 시 주의사항

### 6-1. 파일 포맷 버전 관리

암호화/난독화 파일은 반드시 `type`, `version`을 포함해야 한다.

이유:

- 향후 포맷 교체
- 일반 JSON과의 호환
- 복구 및 마이그레이션 처리

### 6-2. 메타데이터 최소화

암호화 파일 바깥에 남는 `metadata`는 최소화해야 한다.

예:

- 저장 시각
- 내부 포맷 버전
- 사건번호는 필요 여부를 재검토

이름, 가족관계, 생년월일 같은 평문 메타데이터는 외부에 남기지 않는 것이 바람직하다.

### 6-3. 기존 파일과의 공존

기존 JSON 저장과 보호 저장은 당분간 병행하는 것이 좋다.

예:

- `일반 저장`
- `보호 저장(실험)`

### 6-4. 경고 문구 필요

옵션 A 또는 B를 도입할 경우 UI에서 다음을 명확히 밝혀야 한다.

- 일반 열람 방지용이다
- 완전한 보안을 보장하지 않는다

## 7. 향후 작업 후보

1. 보호 저장 기능의 정책 결정
   - A / C 중 어떤 방향을 우선 검토할지 결정

2. 파일 포맷 초안 작성
   - type
   - version
   - metadata
   - payload/ciphertext

3. 저장/불러오기 UI 시안
   - 일반 저장 vs 보호 저장
   - 파일 열기 실패 시 메시지

4. 환경 제약 조사
   - 현재 실행 환경에서 OS 계정 종속 암호화가 가능한지 확인

## 8. 현재 결론

현재 요구를 가장 정직하게 해석하면:

- `메모장에서 바로 읽히지 않게`는 가능
- `비밀번호 없이 자동으로 열리게`도 부분적으로 가능
- 그러나 `강한 개인정보 보호`까지 기대하려면 OS 계정 종속 방식이 필요하다

따라서 현 시점 권고는 다음과 같다.

- 단기: 전용 포맷/난독화 검토
- 장기: OS 계정 종속 암호화 검토
- 비밀번호 없는 완전 보안은 불가능하다는 점을 문서에 명시
