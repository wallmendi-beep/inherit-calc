# 선사망 배우자와 호주상속 가산 규칙

## 1. 핵심 결론

- 선사망 배우자는 상속권이 없는 자일 뿐, 원칙적으로 `피대습상속인`이 아니다.
- 따라서 `배우자 선사망`은 곧바로 `대습상속 개시`로 연결되면 안 된다.
- 배우자 선사망을 근거로 그 배우자 계통에 `호주상속 5할 가산`을 주면 안 된다.

## 2. 김말수-옥태진형 케이스 정리

- `옥태진`이 `김말수`보다 먼저 사망했다면, 김말수 상속에서 옥태진은 배우자 상속인이 아니다.
- 이 경우 옥태진은 `피대습자`가 아니라 `선사망으로 상속권이 소멸한 배우자`이다.
- 따라서 `옥인철이 옥태진 자리를 대신하여 대습상속한다`는 구조로 보면 안 된다.

## 3. 판례·선례형 가산과의 구분

다음 조건이 모두 충족될 때만 판례·선례형 가산 검토 대상으로 본다.

1. 문제되는 인물이 본래 상속인이 될 수 있는 `직계비속 또는 형제자매` 계통일 것
2. 그 인물이 `상속개시 전 사망 또는 결격`으로 빠졌을 것
3. 현재 분배가 그 인물의 `대습상속` 단계일 것
4. 가산 주장 대상자가 `피대습자로부터 실제 호주를 승계한 자`일 것
5. 해당 시점 법리가 `호주상속 가산`을 허용하는 시기일 것

배우자 선사망은 위 1번부터 충족하지 못하므로, 원칙적으로 이 필터를 통과하지 못한다.

## 4. 엔진 반영 원칙

엔진은 아래 규칙을 따라야 한다.

1. `wife/husband/spouse`가 선사망(`isPre`)인 경우:
   - 해당 배우자 본인의 상속분은 `0`
   - 설명은 `피상속인보다 먼저 사망`
   - 그 배우자의 하위 자녀를 근거로 `대습상속 단계`를 열지 않음

2. `대습상속 트리거`는 원칙적으로:
   - `직계비속`
   - `형제자매`
   - 법문상 허용되는 `결격/상실 배우자 대습`
   로 한정한다.

3. `배우자 선사망`과 `대습 호주가산`은 별도 문제로 취급한다.

## 5. 구현 체크포인트

- `shouldSkipHeir()`에서 선사망 배우자를 일반 선사망자와 동일하게 처리하지 말 것
- `isSubstitutionTrigger()`에서 배우자 선사망은 제외할 것
- `canApplyHojuBonus()`에 들어가기 전에 `피대습자 자격`을 먼저 필터링할 것
- 테스트 케이스에 `선사망 배우자 + 자녀 존재` 사례를 추가할 것

## 6. 실무 해석 메모

- 실무상 일부는 `원래 남편이 살아 있었다면 받을 상속분` 논리로 가산을 넓게 보려 한다.
- 그러나 이 논리는 `대습상속 요건`과 `배우자 선사망 탈락`을 혼합할 위험이 있다.
- 본 계산기는 보수적으로, `대습상속 성립 여부`를 먼저 통과한 경우에만 가산 판단으로 넘어간다.
