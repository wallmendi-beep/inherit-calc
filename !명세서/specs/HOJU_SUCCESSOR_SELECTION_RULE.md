# 원호주상속인 선택 규칙

작성일: 2026-04-13

## 목적

컴퓨터가 장남/차남, 출생순위, 선사망 형제 간 우선순위를 자동 추정하지 않도록 한다.

대신 각 피상속인 탭에서
`1차 상속인 중 원래 호주상속을 받을 사람`
을 사용자가 직접 1명 지정하게 하여,
대습 호주가산 판단의 전제 사실을 명확히 고정한다.

## 핵심 원칙

1. 시스템은 `장남 여부`를 자동 판정하지 않는다.
2. 시스템은 `1차 상속인 중 원호주상속인`만 사용자 확인값으로 받는다.
3. 이 값이 없으면 대습 호주가산 판단을 열지 않는다.
4. 이미 사망한 상속인도 원호주상속인으로 지정할 수 있어야 한다.

## 가이드 문구 최종안

기본 안내:

- `{피상속인명}은 호주입니다. 1차 상속인 중 원호주상속인 1명을 지정하세요.`
- `이미 사망한 사람도 지정할 수 있습니다.`

미지정 경고:

- `{피상속인명}의 1차 상속인 중 원호주상속인이 지정되지 않았습니다.`
- `대습 호주가산을 검토하려면 1명을 먼저 지정하세요.`

잘못된 지정 경고:

- `원호주상속인은 1차 상속인 중에서만 지정할 수 있습니다.`

중복 지정 경고:

- `{피상속인명} 탭에는 원호주상속인을 1명만 지정할 수 있습니다.`

## 체크 UI 위치

위치 원칙:

1. 상단 헤더가 아니라 `입력 탭`의 해당 피상속인 카드 안에 둔다.
2. `호주` 체크와 분리된 별도 의미로 표시한다.
3. 사람 행 단위에서 선택하되, 현재 탭의 1차 상속인에게만 노출한다.

권장 배치:

- `상태` 열과 별도
- `특수조건 및 가감산` 열 안 또는 그 왼쪽 보조 열
- 라벨: `원호주상속인`

권장 UI 형태:

- 체크박스 또는 라디오형 단일선택
- 같은 탭에서는 1명만 선택 가능
- 이미 사망한 사람도 선택 가능
- 후속 대습 라인(손자, 며느리 등)에는 직접 노출하지 않음

## 데이터 모델

필요 필드:

- 링크 단위 필드 `isPrimaryHojuSuccessor: boolean`

저장 위치 원칙:

- 사람(person) 자체 속성이 아니라
- `부모 -> 자녀` 관계 링크의 속성으로 저장

이유:

- 어떤 사람은 A의 상속에서는 원호주상속인일 수 있지만
- B의 상속에서는 아닐 수 있으므로
- person 전역 속성으로 두면 안 된다.

## 엔진 조건 연결 방식

직접 호주가산과 대습 호주가산을 분리한다.

### 1. 직접 호주가산

다음 조건을 모두 만족할 때만 검토:

1. 현재 분배 단계의 피상속인이 호주였음
2. 현재 분배 단계의 법시기가 호주가산 검토 대상 시기임
3. 해당 상속인이 `isPrimaryHojuSuccessor === true`
4. 해당 상속인이 아들 계통임
5. 실무해석 모드 또는 허용된 규칙 모드임

### 2. 대습 호주가산

다음 조건을 모두 만족할 때만 검토:

1. 현재 분배 단계에서 대습상속이 실제 발생함
2. 피대습자가 `isPrimaryHojuSuccessor === true` 였음
3. 피대습자는 아들 계통임
4. 대습상속인 중 실제로 호주를 잇는 사람이 있음
5. 그 사람에게만 대습 호주가산을 검토함

즉 단순히
- `isHoju === true`
- `아들`
- `선사망`

만으로는 대습 호주가산을 열지 않는다.

## 적용 효과

이 규칙을 적용하면:

1. 김혁조 사례처럼 `김일상이 원호주상속인인지`를 사용자 확인값으로 고정할 수 있다.
2. 장남/차남 자동판정 오류를 피할 수 있다.
3. 선사망 장남의 아들(장손) 대습 같은 사례에서만 대습 호주가산을 좁게 열 수 있다.
4. 근거 없는 `가산 배제 가능` 문구를 사람별 비고에 남발하지 않게 된다.

## 구현 순서

1. 관계 링크에 `isPrimaryHojuSuccessor` 추가
2. 입력 탭 1차 상속인 행에 단일선택 UI 추가
3. 가이드 패널에 미지정 경고 추가
4. 엔진에서 직접/대습 호주가산 조건 분리
5. 사람별 비고는 실제 영향자에게만 표시
