# 취득경로 계통 트리 및 사건 검토 UX 보강 계획

작성일: 2026-05-01

## 배경

김혁조 사건처럼 선사망, 후사망, 대습상속, 재상속, 차순위 상속이 겹치는 사건에서는 사용자가 AI 입력 결과를 그대로 받은 뒤 가족관계와 사건 흐름을 아직 충분히 파악하지 못한 상태일 수 있다.

현재 UI는 계산 결과 자체는 제공하지만, 일부 화면에서 다음 문제가 남아 있다.

- 사건 검토 카드의 문구가 중복되어 좁은 카드에서 핵심 흐름을 읽기 어렵다.
- 피상속인 카드가 상속인 카드와 같은 디자인 언어를 쓰지 않아 사건 그래프의 시각적 리듬이 어긋난다.
- 계모자 관계 확인 가이드는 전제 관계를 설명하지 않아 사용자가 왜 해당 인물을 확인해야 하는지 이해하기 어렵다.
- 계모자 관계 확인 가이드의 이동 대상이 "빠진 후보자"의 실제 위치와 어긋나, 버튼을 눌러도 후보자가 보이지 않을 수 있다.
- 취득경로 탭은 최종 상속인별 카드/표와 1차 계통 그룹만 제공하므로, 지분이 2차, 3차 사건을 거쳐 이동한 흐름을 한눈에 보기 어렵다.

## 목표

이번 수정의 목표는 계산 로직을 바꾸지 않고, 이미 계산된 `calcSteps`, `finalShares`, `warnings`, `smartGuides`를 더 이해하기 쉬운 사용자 경험으로 표현하는 것이다.

구체적인 목표는 다음과 같다.

1. 사건 검토 카드의 재상속/대습상속 문구를 간결하게 정리한다.
2. 사건 그래프의 피상속인 카드를 상속인 카드와 같은 구조로 맞춘다.
3. 계모자 관계 확인 가이드가 가족관계 전제를 먼저 설명하도록 문구를 바꾼다.
4. 계모자 관계 확인 가이드의 이동 대상과 버튼 의미를 사용자가 실제로 확인해야 할 위치에 맞춘다.
5. 취득경로 탭에 기존 카드뷰/표뷰를 유지한 채 `계통 트리` 보기를 추가한다.
6. 계통 트리에서는 원 피상속인의 지분이 1차, 2차, 3차 사건을 거쳐 최종 생존 상속인에게 도달하는 흐름을 사건 단위로 표시한다.

## 비목표

이번 작업에서는 다음을 하지 않는다.

- 법정상속 계산 공식 자체를 변경하지 않는다.
- 기존 카드뷰와 표뷰의 의미를 계통 트리로 대체하지 않는다.
- 기존 `PathView`의 검산용 표 구조를 크게 바꾸지 않는다.
- 새로운 법리 판단을 추가하지 않는다. 기존 엔진이 산출한 사건과 분배 결과를 시각화한다.

## 수정 범위

### 1. 사건 검토 카드 문구 정리

대상: `src/components/TreePanel.jsx`

현재 취득자 설명 카드에서 사망일이 상단 이름 줄과 하단 연결 문구에 중복 표시된다.

변경 목표:

```text
구수명
(처)
처 감산 적용 → 최종 2/15 취득
└ 1990년 7월 3일 사망 · 재상속 개시
```

원칙:

- 이름 줄에는 사망일을 표시하지 않는다.
- 하단 연결 문구에서만 사망일과 `재상속 개시` 또는 `대습상속 개시`를 표시한다.
- `다음 사건으로 이어짐` 문구는 제거한다.

### 2. 피상속인 카드 디자인 통일

대상: `src/components/TreePanel.jsx`

현재 `PersonNodeCard`는 `node.isRoot`일 때 별도 구조를 사용한다.

변경 목표:

```text
피상속인
망 구수명                       1/1
1990년 7월 3일 사망
```

원칙:

- 피상속인도 상속인 카드처럼 좌측 라벨/이름, 우측 지분 구조를 사용한다.
- 사망일은 하단 보조 정보로 표시한다.
- 호주인 경우 하단에 `호주` 정보를 함께 표시한다.
- 카드 폭과 높이는 상속인 카드와 최대한 같은 리듬을 유지한다.

### 3. 계모자 관계 확인 가이드 문구 개선

대상: `src/hooks/smartGuideHelpers.js`

현재 문구:

```text
계모자 관계 확인 — [조상옥]이 [조창제]의 자녀로 입력됨. 구법상 [윤숙자] 사건의 상속인에 포함될 수 있습니다.
```

변경 목표:

```text
계모자 관계 확인 — [윤숙자]의 배우자 [조창제]에게 자녀 [조상옥]이 입력되어 있습니다. [조상옥]은 [윤숙자]와 계모자 관계일 수 있으므로, 구법상 [윤숙자] 사건의 상속인 포함 여부를 확인해 주세요.
```

원칙:

- AI 자동 입력 후 사용자가 관계를 모를 수 있으므로, 부부 관계와 자녀 관계를 먼저 설명한다.
- "포함될 수 있습니다"로 끝내지 않고 "포함 여부를 확인해 주세요"로 사용자의 행동을 명확히 한다.

### 4. 계모자 관계 확인 가이드 이동 대상 보정

대상:

- `src/hooks/smartGuideHelpers.js`
- `src/hooks/useSmartGuide.js`
- `src/components/SmartGuidePanel.jsx`
- 필요 시 `src/App.jsx`

문제:

- 가이드는 윤숙자 사건에 빠진 조상옥을 감지하지만, 버튼은 윤숙자 사건으로 이동한다.
- 조상옥은 윤숙자 사건에 아직 없으므로 사건을 열어도 보이지 않는다.

변경 목표:

- 가이드에는 윤숙자 사건 탭과 조상옥의 실제 입력 위치를 모두 담는다.
- 기본 확인 이동은 조상옥이 실제로 입력된 위치, 즉 조창제 하위 자녀 위치를 보여준다.
- 버튼 라벨은 상황에 맞게 `위치 보기`, `입력에서 확인`, 또는 `조상옥 위치 보기`처럼 의미를 분명히 한다.
- 필요하면 윤숙자 사건으로 이동하는 보조 행동은 나중에 별도로 둔다.

### 5. 취득경로 탭에 계통 트리 보기 추가

대상:

- `src/App.jsx`
- `src/components/AcquisitionPanel.jsx`
- 신규 `src/components/LineagePathTree.jsx` 또는 유사 컴포넌트

현재 보기:

```text
[카드로 보기] [표로 보기]
```

변경 후:

```text
[카드로 보기] [표로 보기] [계통 트리]
```

원칙:

- 기존 카드뷰와 표뷰는 유지한다.
- 기본값은 기존과 동일하게 `card`로 둔다.
- 계통 트리는 설명용 화면으로 추가한다.
- 인쇄 정책은 우선 기존 표뷰 출력 규칙을 유지하고, 계통 트리 인쇄 포함 여부는 구현 후 화면 밀도를 보고 결정한다.

### 6. 계통 트리 표시 규칙

계통 트리는 `calcSteps`를 사건 노드로 보고 연결한다.

각 사건 노드에는 다음을 표시한다.

- 피상속인 이름
- 사망일
- 해당 사건에 유입된 지분: `step.inN / step.inD`
- 복수 유입이 있으면 `mergeSources` 또는 `sourceBreakdown` 기반으로 유입 출처를 표시
- 해당 사건의 분배 결과: `step.dists`

각 분배 노드에는 다음을 표시한다.

- 상속인 이름
- 관계
- 취득 지분: `dist.n / dist.d`
- 감산, 가산, 균분 등 modifier
- 생존자는 최종 취득자로 표시
- 사망자는 하위 사건이 있으면 `재상속 개시` 또는 `대습상속 개시`로 이어 붙인다

형제자매 상속 표시 원칙:

- 별도 "형제자매 계통"으로 분리하지 않는다.
- 해당 지분이 발생한 직계 흐름 안에서 `차순위 분배` 또는 관계 라벨로 표시한다.
- 사용자가 왜 방계로 지분이 이동했는지 흐름 안에서 이해할 수 있게 한다.

예시:

```text
망 김혁조 상속분 1/1
├─ 김명수 1/15 취득
│  └─ 1972년 사망 · 재상속 개시
│     ├─ 정문자 1/45 취득
│     ├─ 김명남 1/45 취득
│     └─ 김상옥 1/45 취득
│
├─ 구수명 2/15 취득
│  └─ 1990년 사망 · 재상속 개시
│     ├─ 김명남 2/165 취득
│     ├─ 김명수 계통 2/165
│     └─ 김일상 계통 ...
│
└─ 김일상 2/5 취득
   └─ 사망 · 재상속 개시
      └─ ...
```

## 구현 순서

1. 기존 변경 요청 4개 반영 상태를 점검한다.
   - `isSpouseRelation` 통일
   - `grouped-direct-missing` 워프 위치
   - 취득경로 카드/표 토글 및 `PathView` 분리
   - SummaryPanel 토글 제거
2. 사건 검토 카드 문구를 간결화한다.
3. 피상속인 카드 디자인을 상속인 카드와 통일한다.
4. 계모자 관계 가이드 문구와 이동 대상을 보정한다.
5. 계통 트리 데이터 빌더를 만든다.
6. `LineagePathTree` 컴포넌트를 추가한다.
7. 취득경로 탭 토글에 `계통 트리`를 추가한다.
8. 테스트를 추가하거나 기존 테스트 문구를 갱신한다.
9. `npm run test`를 통과시킨다.
10. `npm run build`를 통과시킨다.

## 검증 기준

- `npm run test` 전체 통과
- `npm run build` 성공
- 취득경로 탭 기본값은 `card`
- 기존 카드뷰와 표뷰의 내용은 유지
- `계통 트리`에서 1차 계통뿐 아니라 후속 재상속/대습상속 흐름이 표시됨
- 계모자 관계 확인 가이드가 전제 관계를 설명함
- 계모자 관계 확인 가이드 버튼을 눌렀을 때 후보자의 실제 위치를 확인할 수 있음
- 사건 검토 카드에서 사망일과 재상속/대습상속 문구가 중복되지 않음
- 피상속인 카드가 상속인 카드와 같은 카드 구조로 보임
