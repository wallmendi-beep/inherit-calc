# 토글 상태 판정 규칙 2026-04-13

본 문서는 입력 탭의 좌측 상태 토글이 언제 `ON`, 언제 `OFF`로 보여야 하는지를 고정하는 SSOT다.

## 1. 기본 원칙

- 상태 토글은 단순히 `isExcluded` 원본값만 보여주지 않는다.
- 현재 상속 단계에서 하위 상속인이 실제로 입력되었는지까지 함께 본다.
- 하위 상속인이 생기면, 원칙적으로 지분 전이 경로가 살아 있으므로 `ON`으로 복구한다.

## 2. ON으로 복구하는 경우

아래 조건을 모두 만족하면 기본적으로 `ON`으로 본다.

- 하위 상속인이 1명 이상 존재한다.
- 배우자 선사망이 아니다.
- 아래 `OFF 유지 예외`에 해당하지 않는다.

대표 사례:

- 선사망 아들인데 자녀가 입력되어 대습상속이 열림
- 후사망자에게 배우자/자녀가 입력되어 재상속 경로가 생김
- stale `predeceased` 또는 과거 파생 `isExcluded` 값이 남아 있었지만, 현재는 하위 상속인이 존재함

## 3. OFF 유지 예외

하위 상속인이 있어도 아래 사유는 계속 `OFF`로 유지한다.

- `lost`
- `disqualified`
- `remarried`
- `renounce`
- `blocked_husband_substitution`
- 선사망 배우자

즉, 하위 상속인이 있다는 사실만으로 포기/결격/재혼/사위 대습상속 불가를 무시하면 안 된다.

## 4. 화면 표시 규칙

- 토글의 시각 상태는 `현재 계산 경로가 살아 있는지`를 우선 반영한다.
- 따라서 `isExcluded === true`가 저장돼 있어도, 하위 상속인이 생긴 선사망 자녀/형제자매는 화면에서 `ON`으로 보여야 한다.
- 반대로 법리상 차단 사유가 분명한 경우는 하위 상속인이 있어도 `OFF`를 유지한다.

## 5. 복원 경로 규칙

`vault -> tree` 복원 단계에서도 위 규칙을 동일하게 적용한다.

- 화면에서만 `ON`으로 보이고 내부 복원에서는 다시 `OFF`로 돌아가면 안 된다.
- 따라서 `vaultTransforms`와 `HeirRow`는 같은 판정 기준을 공유해야 한다.

## 6. 현재 구현 메모

현재 구현은 아래 파일에 반영되어 있다.

- `src/components/HeirRow.jsx`
- `src/utils/vaultTransforms.js`

핵심 구현 의도:

- 하위 상속인이 생긴 뒤에는 stale exclusion을 걷어낸다.
- 다만 `lost/disqualified/remarried/renounce/blocked_husband_substitution/선사망 배우자`는 유지한다.
