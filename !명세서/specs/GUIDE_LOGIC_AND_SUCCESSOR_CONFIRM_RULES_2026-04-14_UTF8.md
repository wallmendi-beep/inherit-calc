# GUIDE LOGIC AND SUCCESSOR CONFIRM RULES 2026-04-14

## 목적
- 스마트 가이드가 왜 뜨는지, 어떤 입력 상태에서 어떤 안내가 우선 노출되는지 명확히 고정한다.
- `불러오기`가 필요한 경우와 `후속 상속인 없음 확정` 버튼으로 끝내도 되는 경우를 구분한다.
- 입력 탭 빈 슬롯 안내, 스마트 가이드, 요약 경고가 서로 모순되지 않도록 단일 원칙을 만든다.

## 기본 철학
- 데이터 입력 탭은 사실 입력과 상태 확정의 공간이다.
- 계산 상세/결과 탭은 법리 판단과 지분 결과를 보여주는 공간이다.
- 가이드는 사용자를 겁주기보다, “지금 무엇을 입력/확정해야 하는지”를 짧게 알려주는 조수 역할을 맡는다.

## 가이드 패널 UX 원칙
- 가이드는 기본적으로 우측 패널에 펼쳐진다.
- 상단 `가이드` 버튼은 열기 버튼이 아니라 펼침/접힘 토글로 이해되도록 동작한다.
- 접힌 상태에서는 우측 상단 플로팅 `가이드` 버튼이 남는다.
- 접힌 상태에서 필수 검토/무결성 경고/감사 이슈가 있으면 플로팅 버튼은 앰버 톤으로 바뀐다.

## 빈 슬롯 안내 우선순위
입력 탭에서 현재 노드의 `heirs.length === 0`일 때는 아래 우선순위로 한 개의 안내만 노출한다.

1. `successorStatus`가 이미 확정되어 있으면:
   - 제목: `후속 상속인 없음이 확정되었습니다.`
   - 설명: `이 단계는 추가 상속인 없이 진행하도록 확인된 상태입니다.`

2. 현재 탭에 `inheritance-cycle` 이슈가 있으면:
   - 제목: `순환 참조 오류를 먼저 정리해 주세요.`
   - 설명은 해당 이슈 텍스트를 그대로 사용한다.

3. 루트 기본정보가 비어 있으면:
   - 제목: `기본정보를 먼저 입력해 주세요.`

4. 선사망 상속인이면:
   - 제목: `선사망 상속인은 대습상속인 입력이 필요합니다.`
   - 설명: `배우자 또는 직계비속을 입력하지 않으면 계산에서 제외됩니다.`

5. 구법 + 아들/남편 단계에서 `isHoju === false`이면:
   - 제목: `OOO은(는) 비호주로 설정되어 있습니다.`
   - 설명: `특별한 사정이 없다면 호주 여부를 다시 확인한 뒤 1차 상속인을 입력해 주세요.`

6. 구법 + 루트/아들/남편 단계면:
   - 제목: `이 상속 단계는 호주상속 가산 검토가 필요합니다.`
   - 설명: `상속인 불러오기 또는 수동 입력을 먼저 진행한 뒤, 현재 탭에서 호주 여부를 확인하세요.`

7. 그 외 사망자 빈 슬롯:
   - 제목: `아직 직접 입력된 후속 상속인이 없습니다.`
   - 설명: `필요한 상속인이 있다면 추가해 주세요. 입력하지 않으면 법정 순위에 따라 자동 분배가 진행될 수 있습니다.`

## 호주상속 판단을 위해 불러오기가 필요한 경우
다음 조건을 동시에 만족하면, 이 단계는 사실상 `불러오기` 또는 수동 입력이 필요하다.

- 적용 시기가 `1991` 이전
- 현재 탭이 `root` 또는 관계가 `son`, `husband`
- 이 단계에서 호주상속 가산 가능성이 있음

이 경우 앱은:
- 입력 탭 빈 슬롯 안내로 `호주상속 가산 검토가 필요합니다`를 띄운다.
- 스마트 가이드에서는 같은 취지의 호주 검토 가이드를 띄운다.
- 사용자는 먼저 `불러오기` 또는 수동 입력으로 1차 상속인을 만든 뒤,
  - 피상속인 본인의 `호주/비호주`
  - 1차 상속인의 `호주상속/재산상속`
  를 확인해야 한다.

## 왜 윤종옥 같은 경우는 불러오기가 필요한가
- 윤종옥 단계에서는 단순히 “사망자 여부”만이 아니라,
  - 윤종옥 본인이 호주였는지
  - 1차 상속인 중 누가 호주상속을 받는지
  를 함께 확인해야 한다.
- 따라서 `호주상속 판단이 필요한 중간 사건`에서는 후속 상속인 불러오기가 사실상 필요하다.

## 후속 상속인 없음 확정 버튼 규칙
입력 탭 빈 슬롯 안내 아래에는 상황별 `마감 버튼`을 둔다.

### 1. 배우자 단계
- 버튼: `추가 상속인 없음`
- 저장값: `successorStatus = confirmed_no_additional_heirs`
- 의미: 이 단계에서 더 입력할 후속 상속인이 없음을 사용자가 확인했다.

### 2. 선사망 자녀/형제자매 단계
- 버튼: `대습상속인 없음`
- 저장값: `successorStatus = confirmed_no_substitute_heirs`
- 의미: 배우자나 직계비속이 없어 대습상속이 없음을 확인했다.

### 3. 기타 사망자 단계
- 버튼: `직계비속·배우자 없음`
- 저장값: `successorStatus = confirmed_no_spouse_descendants`
- 의미: 직계비속과 배우자가 없어 차순위 상속으로 넘어가도 된다는 점을 확인했다.

### 동작 원칙
- 버튼을 누르면 빈 슬롯 안내는 즉시 `후속 상속인 없음이 확정되었습니다.` 상태로 바뀐다.
- 같은 사람에 대한 `후속 상속인 누락` 스마트 가이드도 더 이상 만들지 않는다.
- 요약 탭의 “사망자인데 하위상속인이 비어 있음” 경고에서도 제외한다.
- 나중에 `상속인 추가` 또는 `불러오기`를 하면 해당 사람의 `successorStatus`는 자동으로 초기화한다.
- `확정 해제` 버튼으로 사용자가 다시 빈 상태 경고 흐름으로 돌아갈 수 있다.

## 스마트 가이드의 목록형 묶음 규칙
같은 종류의 누락 가이드는 사람별 경고를 줄줄이 띄우지 않는다.

### 1. 선사망 상속인 대습 누락
- 개별 경고를 만들지 않는다.
- 부모 단계 기준으로 묶어서 한 개만 만든다.
- 예:
  - `선사망 상속인 중 대습상속인이 입력되지 않은 사람이 있습니다: [윤우경], [윤우철].`

### 2. 일반 후속 상속인 직접 입력 누락
- 개별 경고를 만들지 않는다.
- 부모 단계 기준으로 묶어서 한 개만 만든다.
- 예:
  - `직접 입력된 후속 상속인이 없는 사람이 있습니다: [정차윤], [OOO].`

### 3. 중복 제거 원칙
- 앱은 `type + text`를 기준으로 한 번 더 dedupe 한다.
- 같은 뜻의 가이드는 `uniqueKey`가 달라도 하나만 남긴다.

## 실제 상태 저장 위치
- `successorStatus`는 사람 정보(person fact)에 저장한다.
- 즉 특정 부모 링크 전용이 아니라, 해당 사건 탭에서 그 사람의 후속 상속 상태를 나타내는 입력 확인값으로 본다.
